# Indus Image Generator - Project Context

Comprehensive reference for agents and developers working on this codebase. Read this before making changes.

---

## 1. Project Overview

Indus Image Generator is a full-stack AI image generation app built on **React 18 + Tailwind CSS v4** (frontend) and **Supabase Edge Functions / Deno + Hono** (backend). It connects to **RunPod Serverless** to execute **ComfyUI** workflows using a fine-tuned **Qwen Image 2512** diffusion model with a custom **Indus LoRA** style.

The app is a chat-based interface: the user types a prompt, the system wraps it in a style template, submits a ComfyUI workflow to a RunPod Serverless endpoint, polls for completion, and displays the resulting image inline. **No dedicated GPU pods are managed** — the serverless endpoint handles cold starts, scaling, and shutdown automatically.

**Tech stack:**
- Frontend: React 18, Tailwind CSS v4, Vite, lucide-react icons
- Backend: Deno, Hono web server (Supabase Edge Functions)
- GPU: RunPod **Serverless** (REST API — `/v2/{endpoint_id}/run`, `/status/{job_id}`, `/health`)
- Storage: Supabase KV store (`kv_store_9f1ace44` table)
- Fonts: Inter (UI text), JetBrains Mono (metadata/technical text)
- Animation: Pure CSS `@keyframes` + imperative Web Animations API (canvas-based animation for GenerationVisual). **Do NOT use `motion/react` (Framer Motion)** — it causes a dynamic import crash in this environment despite being in package.json.

## 2. Architecture

```
Frontend (React SPA)
    |
    | HTTPS (Bearer token auth via publicAnonKey)
    v
Supabase Edge Function (Hono server)
    |
    |--- RunPod Serverless API (POST /run, GET /status/{id}, GET /health)
    |--- Supabase KV Store (workflows, config, job metadata)
```

All routes are prefixed with `/make-server-9f1ace44`. The frontend constructs the base URL as:
```
https://${projectId}.supabase.co/functions/v1/make-server-9f1ace44
```

The `projectId` and `publicAnonKey` are imported from `/utils/supabase/info.tsx` (autogenerated, do not edit).

### Serverless vs. Dedicated Pods (Historical Context)

The previous architecture managed dedicated RunPod GPU pods via the GraphQL API — discovering, creating, starting, stopping, and auto-stopping pods after idle timeouts. This has been **completely replaced** with RunPod's Serverless API. The server now acts as a stateless middleman: it builds the workflow, submits it to the serverless endpoint, and polls for the result. Pod lifecycle management code has been removed entirely.

## 4. Frontend Components

### 4.1 App.tsx (Root Orchestrator)

**State owned:**
- `messages: ChatMessage[]` — All chat messages (user prompts + bot responses/images)
- `isGenerating: boolean` — Whether a generation is in progress
- `generationPhase: string` — Current status text during generation
- `generationProgress: number` — 0-100 progress percentage
- `selectedStyle: StyleKey` — Currently selected style ("Indus", "PhonePe", etc.)

**Flow:** User sends prompt via `ChatPanel.onSendMessage` -> `handleSendMessage` adds user message to state, calls `generateImage()` from api-service with phase/progress callbacks, then appends bot message (image or error) on completion.

**Layout:** Full-screen horizontal flex: `StyleSidebar` (56px fixed) + `ChatPanel` (flex-1).

### 4.2 ChatPanel (`chat-panel.tsx`)

The main UI surface. Renders:
- **Header:** Style name + StatusIndicator
- **Floating toolbar:** Generate (sparkles icon), Save All (FolderDown with count badge), Reset (Trash2)
- **Message list:** Scrollable area with user/bot messages
- **Generation loader:** Shows `GenerationVisual` component during active generation
- **Input area:** Textarea + attachment button + dimension selector dropdown + send button
- **Modals:** Reset confirmation, fullscreen image viewer

**CSS Animations (injected via `<style>` tag, NOT via motion/react):**
- `cp-msg-enter` — Message slide-up entrance
- `cp-loader-enter` — Loader slide-up (0.35s delay)
- `cp-send-pulse` — Send button press feedback
- `cp-modal-enter` — Modal scale+slide entrance
- `cp-modal-overlay` — Modal backdrop fade

### 4.3 GenerationVisual (`generation-visual.tsx`)

A self-contained 380x380px canvas animation that plays during image generation. Entirely imperative (DOM manipulation + Web Animations API in a `useEffect`). No React re-renders during animation — only `progressRef` is synced.

**Props:** `phase: string`, `progress: number (0-100)`

**Container matching:** The 380x380 container uses `rounded-xl overflow-hidden border border-white/5` and `background: rgba(22, 22, 32, 0.9)` — this MUST match the image message containers in chat-panel exactly.

### 4.4 StatusIndicator (`status-indicator.tsx`)

Polls `fetchStatus()` every 15 seconds. Displays a colored dot + label:
- Green (pulsing): "Serverless Ready"
- Amber (pulsing): "Starting..."
- Gray: "Checking..."

### 4.5 SettingsModal (`settings-modal.tsx`)

Three-tab modal:
- **Status tab:** Connection test, serverless endpoint health (worker counts), refresh
- **Workflows tab:** View built-in/custom workflow status, upload custom overrides
- **Config tab:** Workflow parameters (checkpoint, sampler, scheduler, steps, CFG)

## 6. Frontend API Client (`api-service.ts`)

Central module for all backend communication. All calls use `apiFetch<T>()` helper with:
- AbortController timeout (default 30s)
- Authorization: `Bearer ${publicAnonKey}`
- JSON content type

### Key functions

| Function | Route | Purpose |
|----------|-------|---------|
| `fetchStatus()` | GET `/comfyui/status` | Serverless endpoint health |
| `getWorkflowConfig()` | GET `/comfyui/config` | Read workflow params |
| `saveWorkflowConfig()` | POST `/comfyui/config` | Save workflow params |
| `uploadWorkflow()` | POST `/comfyui/workflow/upload` | Upload custom workflow JSON |
| `getWorkflowInfo()` | GET `/comfyui/workflow` | Check if custom workflow exists |
| `generateImage()` | POST `/comfyui/generate` + GET `/comfyui/status/:jobId` | Full generation flow |
| `testConnection()` | GET `/comfyui/status` | Latency test |

### Generation flow (frontend)

`generateImage()` is now simpler without pod auto-start recovery:

1. POST to `/comfyui/generate` with prompt, dimensions, style, reference image
2. Server returns `{ job_id, seed }`
3. `pollForResult()` polls `/comfyui/status/:jobId` every 2s (up to 150 attempts = ~5 min)
4. Phase callbacks update GenerationVisual with descriptive messages
5. Return `GenerateResponse` with image data or error

**Cold start handling:** The serverless worker may take 30-60s on first request. The polling loop's phase messages account for this with messages like "Waking up GPU...", "Loading models...", etc.

## 7. Backend Server (`index.tsx`)

Hono web server with CORS (`origin: *`) and console logger. All routes under `/make-server-9f1ace44/`.

### Routes

| Method | Path | Handler |
|--------|------|---------|
| GET | `/health` | Simple health check |
| GET | `/comfyui/status` | Serverless endpoint health (`/v2/{endpoint}/health`) |
| GET | `/comfyui/config` | Read workflow config from KV |
| POST | `/comfyui/config` | Save workflow config to KV |
| POST | `/comfyui/workflow/upload` | Store custom workflow JSON in KV |
| GET | `/comfyui/workflow` | Check if custom workflow exists |
| POST | `/comfyui/generate` | Build workflow + submit to serverless |
| GET | `/comfyui/status/:jobId` | Poll serverless job status |
| GET | `/comfyui/debug/check` | Debug info (API key, endpoint ID, health) |

### Removed routes (from pod architecture)
- `POST /comfyui/pod/start` — No longer needed
- `POST /comfyui/pod/stop` — No longer needed
- `GET /comfyui/pods` — No longer needed
- `POST /comfyui/pod/select` — No longer needed
- `POST /comfyui/workflow/run` — No longer needed

### Generate flow (server-side)

1. Parse request (prompt, width, height, seed, style, reference_image)
2. Build prompt using Indus templates (`buildIndusPrompt` / `buildControlnetPrompt`)
3. Get workflow (KV-custom or embedded), apply parameters via `applyWorkflowParams`
4. For ControlNet: include base64 reference image in the `images` field of the serverless payload
5. POST to RunPod Serverless `/v2/{endpoint}/run` with `{ input: { workflow, images? } }`
6. Store job metadata in KV as `indus_job_{jobId}`
7. Return `{ job_id, seed }`

### Job status polling (server-side)

1. GET RunPod Serverless `/v2/{endpoint}/status/{jobId}`
2. If `COMPLETED`: extract base64 image from `output.image` or `output.images[0].data`, return with metadata
3. If `FAILED` / `CANCELLED` / `TIMED_OUT`: return error, clean up KV
4. If `IN_QUEUE` / `IN_PROGRESS`: return `{ status, completed: false }`

## 8. RunPod Serverless Module (`runpod.tsx`)

### Key Functions

| Function | Purpose |
|----------|---------|
| `getEndpointHealth()` | GET `/v2/{endpoint}/health` — returns worker counts and job stats |
| `submitServerlessJob(workflow, images?)` | POST `/v2/{endpoint}/run` — submits a ComfyUI workflow |
| `pollServerlessJob(jobId)` | GET `/v2/{endpoint}/status/{jobId}` — polls job status |
| `buildIndusPrompt(subject)` | Wraps subject in Indus style template |
| `buildControlnetPrompt(subject)` | Wraps subject in ControlNet style template |
| `getWorkflow(type, params)` | Builds parameterized workflow (KV-custom or embedded) |
| `applyWorkflowParams(workflow, mapping, params)` | Injects prompt/seed/dimensions into workflow nodes |

### Serverless Payload Format

```ts
// Submit
POST /v2/{endpoint_id}/run
{
  "input": {
    "workflow": { /* Full ComfyUI workflow JSON with params applied */ },
    "images": {  /* Optional: for ControlNet reference images */
      "ref_1234567890.png": "base64_image_data..."
    }
  }
}

// Response
{ "id": "job_abc123", "status": "IN_QUEUE" }
```

```ts
// Poll
GET /v2/{endpoint_id}/status/{job_id}

// In progress
{ "status": "IN_PROGRESS" }

// Completed
{
  "status": "COMPLETED",
  "output": {
    "image": "base64_image_data...",
    "seed": 12345,
    "execution_time": 23.4
  }
}
```

### Prompt Templates

**Default (text-to-image):**
```
High-end vector icon of {subject}, smooth glossy plastic material, rounded and beveled edges,
thick soft outline slightly lighter than fill, vibrant controlled gradients with tinted highlights,
soft studio lighting from upper-left, subtle rim light, gentle inner shading for faux depth,
short soft drop shadow beneath object, floating centered composition with slight perspective tilt,
premium modern UI illustration, ultra-clean surface, no texture grain, no realism, solid black background
```

**ControlNet (style transfer):**
```
Flat vector illustration of a {subject}, [same styling directives]...
```

### Embedded Workflows

Two production workflows are embedded directly in code:
- `EMBEDDED_DEFAULT_WORKFLOW` — Text-to-image (Qwen Image 2512 + Indus LoRA)
- `EMBEDDED_CONTROLNET_WORKFLOW` — Image-guided generation (Canny edge + ControlNet)

**Workflow priority:** KV-stored custom workflow > Embedded workflow.

## 12. Critical Constraints & Gotchas

### DO NOT use `motion/react` (Framer Motion)
Despite being installed in package.json, dynamic imports from `motion/react` crash at runtime. All animations use CSS `@keyframes` or imperative Web Animations API.

### Protected files (never edit)
- `/src/app/components/figma/ImageWithFallback.tsx`
- `/pnpm-lock.yaml`
- `/supabase/functions/server/kv_store.tsx`
- `/utils/supabase/info.tsx`

### Server code constraints
- All server code in `/supabase/functions/server/` (no subdirectories)
- Cannot import from outside this directory
- External packages via `npm:` or `jsr:` specifiers
- CORS must be imported from `npm:hono/cors` (NOT `npm:hono/middleware`)
- Logger from `npm:hono/logger` (NOT `npm:hono/middleware`)
- Start server with `Deno.serve(app.fetch)`

### Serverless cold starts
First request to the endpoint after all workers have scaled to zero will incur a 30-60 second cold start. The polling loop's phase messages handle this gracefully. The `/comfyui/status` route reports `pod_status: "ready"` even with 0 workers (since serverless auto-starts on demand).

## 13. Generation Lifecycle (End-to-End)

```
1. User types prompt + selects aspect ratio + (optional) attaches reference image
2. User clicks Send (or presses Enter)
                |
3. ChatPanel.handleSend()
   - Adds user ChatMessage to state
   - Sets isGenerating=true, progress=5
                |
4. App.handleSendMessage() calls generateImage()
                |
5. api-service.generateImage()
   - POST /comfyui/generate with { prompt, width, height, style, reference_image?, mode? }
                |
6. Server: index.tsx /comfyui/generate handler
   a. buildIndusPrompt(subject) or buildControlnetPrompt(subject)
   b. getWorkflow(type, params) → applyWorkflowParams()
   c. (ControlNet) include base64 image in serverless payload
   d. submitServerlessJob(workflow, images?) → RunPod Serverless POST /run
   e. Store job metadata in KV
   f. Return { job_id, seed }
                |
7. Frontend polls pollForResult()
   - GET /comfyui/status/{jobId} every 2s (up to 150 attempts = ~5 min)
   - Phase callbacks update GenerationVisual progress
                |
8. Server: /comfyui/status/:jobId
   - GET RunPod Serverless /status/{jobId}
   - On COMPLETED: extract base64 image from output, return with metadata
                |
9. Frontend receives image
   - Adds bot ChatMessage with image data URL
   - Sets isGenerating=false
   - GenerationVisual unmounts, image message renders in chat
```

## 14. RunPod Serverless Endpoint Setup

To create the serverless endpoint that this app connects to:

1. **Build & push the Docker image** containing ComfyUI + handler.py (see `ServerLessProjectContext.md` for details)
2. In RunPod Console, create a **Template** with:
   - Docker image: your pushed image
   - Container disk: 30GB
   - Network volume mounted at `/runpod-volume` (with all model weights)
3. Create a **Serverless Endpoint** with:
   - GPU: RTX 4090+ (24GB VRAM minimum)
   - Min workers: 0 (scale to zero when idle)
   - Max workers: 3 (or as needed)
   - Idle timeout: 5s
   - Execution timeout: 600s
   - Flash Boot: ON
4. Copy the **Endpoint ID** and add it as a Supabase secret: `RUNPOD_ENDPOINT_ID`

## 15. Style System

The app supports multiple brand styles. Each style has a logo, accent color, and ring color. Only "Indus" is currently enabled; the others have `disabled: true` in STYLE_CONFIG.

Messages store the style they were created with (`msg.style`), and bot avatars are resolved from this stored style.
