# 09 — Authentication

Google OAuth via Supabase, with a hard filter on @phonepe.com emails. Simple.

## Flow

```
User lands on /  (Vercel)
     │
     ▼
<AuthProvider> subscribes to Supabase onAuthStateChange
     │
     ▼
<AuthGate> — renders one of:
     │
     ├── If import.meta.env.DEV          →  <AuthenticatedApp />        (dev bypass, mock user "EL@local.dev")
     ├── If loading                      →  spinner
     ├── If no user                      →  <LoginScreen />
     └── If user                         →  <AuthenticatedApp />
```

## Files

| File | Role |
|---|---|
| `src/app/components/auth-context.tsx` | Context + hook that wraps Supabase's session state |
| `src/app/components/login-screen.tsx` | "Sign in with Google" button |
| `src/app/components/supabase-client.ts` | Creates the supabase-js client |
| `utils/supabase/info.tsx` | Hardcoded `projectId` + `publicAnonKey` (autogenerated, committed) |
| `src/app/App.tsx` → `AuthGate` | Routing decision |

## Provider

Google OAuth is configured on the Supabase project:
- Dashboard: <https://supabase.com/dashboard/project/wbpzkcblgoxqtfrduotd/auth/providers>
- Callback URL: `https://wbpzkcblgoxqtfrduotd.supabase.co/auth/v1/callback`

## Allowed emails

A Supabase Auth hook or RLS/trigger rejects any user whose email doesn't end in `@phonepe.com`. Enforced server-side, not just client-side.

## Local dev bypass

In Vite dev mode (`import.meta.env.DEV === true`), `AuthGate` skips auth entirely and renders `AuthenticatedApp` with a hardcoded user:
```ts
user = { id: "local-dev", email: "EL@local.dev" }
```

This:
- Removes the Google OAuth round-trip for local testing
- Makes all activity / gallery / pod-naming use the email prefix `el`
- Would be disabled automatically in production (Vercel builds have `DEV === false`)

Defined inline in `App.tsx::AuthenticatedApp`:
```tsx
const { user: authUser } = useAuth();
const user = authUser ?? (import.meta.env.DEV
  ? { id: "local-dev", email: "EL@local.dev" }
  : null);
```

## How user identity is used

- **Activity log key**: `activity_{email}_{ts}_{jobId}`
- **Gallery key**: `gallery_{email}_{ts}_{jobId}` (and S3 path `gallery/{emailPrefix}/{jobId}.png`)
- **Pod name**: derived from `email.split("@")[0]` (e.g. `elson`, `rahul`) — so multiple users get separate pods
- **Activity feed user prefix**: same email prefix

## Session management

Pure Supabase. Session tokens are in `localStorage` under the supabase-js default keys. Refresh tokens are auto-rotated by supabase-js.

## Edge function identity

The edge function doesn't currently verify the user's JWT. Every request is authenticated with the `anon` key. `user_id` and `email` are passed as body params by the client and trusted.

**Known gap**: an authenticated @phonepe.com user could impersonate another user by sending a different `email` in the body. For an internal tool it's acceptable, but should be hardened before external exposure. Proper fix would be validating the JWT in the edge function and deriving the user server-side.

## Signing out

Handled by Supabase's `signOut()` method — clears localStorage, triggers `onAuthStateChange` with `session = null`, `AuthGate` re-renders to `<LoginScreen />`. Not currently exposed in the UI (no sign-out button yet).
